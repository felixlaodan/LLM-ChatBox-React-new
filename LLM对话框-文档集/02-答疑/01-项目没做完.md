## 🔍 我们来盘点一下现状（看源码分析）

在你的 `src/lib/api.ts` 中，我们完美实现了原生的 Fetch 和 SSE 读取：
```typescript
while (true) {
  const { done, value } = await reader.read()
  // ...解码...
  fullContent += delta.content
  onUpdate(fullContent, fullReasoning) // ⚠️ 问题出在这里！
}
```

而在你的 `src/hooks/useSubmitMessage.ts` 中，你是这样接收的：
```typescript
await createChatCompletion(apiMessages, (content, reasoningContent) => {
  // ⚠️ 每次大模型吐出一个字，这里就会触发一次 zustand 的 updateLastMessage
  updateLastMessage(content, reasoningContent) 
})
```

### 🚨 现在的隐患（痛点）：
大模型每秒可能吐出 20-50 个 Token。意味着那个 `while` 循环每秒会调用几十次 `onUpdate`。
进而导致你的 Zustand Store 每秒被修改几十次，React 组件（MessageBubble）就会**每秒重渲染几十次**！这就叫“无效重渲染”，在长文本时一定会卡顿。

## 🛠️ 后续我们要怎么把“亮点”变成现实？

为了把你在简历上吹的牛逼（`useRef` 配合自定义 Hook 优化）落地，我们**在第四或第五阶段**，会去改造 `useSubmitMessage.ts`。

**未来的改造思路（剧透）：**

我们会引入 `useRef` 作为内存缓冲区，并引入 `lodash/throttle`（或者手写节流/防抖），代码大概会变成这样：

```typescript
// 提前剧透未来的优化版 useSubmitMessage
import { useRef, useCallback } from 'react'
import throttle from 'lodash/throttle'

export const useSubmitMessage = () => {
  const { updateLastMessage } = useChatStore()
  
  // 1. 🌟 定义一个 useRef 作为文本拼接的“暂存区”（改 ref 不会触发渲染！）
  const bufferRef = useRef({ content: '', reasoning: '' })

  // 2. 🌟 定义一个节流函数，比如每 50ms 才真正去更新一次 Zustand Store
  const throttledUpdate = useCallback(
    throttle(() => {
      updateLastMessage(bufferRef.current.content, bufferRef.current.reasoning)
    }, 50),
    []
  )

  const submitMessage = async (text: string) => {
    // ...前面逻辑不变...

    // 清空上次的 buffer
    bufferRef.current = { content: '', reasoning: '' }

    await createChatCompletion(apiMessages, (content, reasoningContent) => {
      // 3. 🌟 收到 Token 时，只更新 Ref 内存，不触发 React 渲染！
      bufferRef.current.content = content
      if (reasoningContent) bufferRef.current.reasoning = reasoningContent

      // 4. 🌟 调用节流函数。就算这里被 1秒调用 50次，真正执行 updateLastMessage 的也只有 20 次。
      throttledUpdate()
    })
  }
}
```
