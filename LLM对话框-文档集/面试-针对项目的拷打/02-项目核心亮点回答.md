# 面试题：请你讲讲“流式响应与高频重渲染优化”这个项目亮点？

## 💡 回答思路拆解

当面试官让你讲简历上这句话时，他其实是在考察你对 **React 渲染机制** 和 **网络底层 API (SSE/Fetch)** 的理解。

**你的回答分为三个层次：痛点（为什么要做） -> 核心技术方案（怎么做的） -> 细节处理与收益（做得多好）。**

以下是为你准备的满分回答话术：

---

### 第一层：提出痛点（大模型场景下的性能灾难）
“好的。在做这个 AI 聊天应用的时候，我发现大模型的返回方式和传统接口不同。它是流式输出（Streaming）的，也就是后端会像打字机一样，一个 Token 一个 Token 地把数据推给前端。
**这里就产生了一个严重的性能问题**：如果按照传统的 React 写法，每次收到一个新的 Token，我就去调用一次 `setState` 来更新当前的消息内容。由于大模型生成速度非常快（一秒钟可能几十个 Token），这会导致 React 组件**极高频地触发 Re-render（重渲染）**。在长对话或者长文本场景下，页面会变得极其卡顿，甚至连用户的输入框都会被卡住。”

### 第二层：详细讲解解决方案（useRef + 渲染截流）
“为了解决这个问题，我放弃了直接用 State 接收高频数据的做法，设计了一套**‘数据接收’与‘视图渲染’分离**的策略。

具体是这样做的：
1. **网络层处理：** 我没有用 Axios，而是基于原生的 `Fetch API` 封装了请求。拿到 `response` 后，通过 `response.body.getReader()` 获取可读流，然后用一个 `while` 循环不断读取二进制块（Chunk），并解析成字符串。
2. **useRef 暂存（最核心的点）：** 我使用了一个 `useRef` 来作为内存中的**缓冲区**。因为在 React 中，修改 `ref.current` **不会触发组件重渲染**。每次解析出新的 Token，我就把它无声无息地拼接到 `ref.current` 里。
3. **节流更新 State：** 只有当缓冲区积累到一定程度，或者达到了一定的时间间隔（比如我加了一个 50ms 的节流/Throttle 策略，或者利用 React 18 的批量更新特性），我才把 `ref.current` 的完整字符串同步给真正的 React State。

这样一来，我把原本可能 100 次的无效 DOM 渲染，压缩到了十几次，完美解决了卡顿问题。”

### 第三层：补充细节与收益展现（展示工程化思考）
“除了渲染优化，在这个自定义 Hook (`useStream`) 的封装里，我还处理了两个关键细节：
- **第一是粘包和拆包问题：** 因为 SSE 传输时，一条完整的 JSON 有可能被截断在两个数据块里。我用了一个内部变量作为 Buffer，按换行符 `\n\n` 去切割，确保只 `JSON.parse` 完整的数据包，避免了解析报错。
- **第二是请求的中断控制：** 我结合了 `AbortController`。如果用户中途想停止生成，或者切换了对话，我会立刻调用 `.abort()` 掐断数据流，避免内存和带宽的浪费。

**最终的收益是**：重构之后，即便是一次性生成几千字的代码块或者 Markdown 文本，页面的滚动和打字机效果都非常丝滑，FPS 稳稳保持在 60 左右。”

---

### 🧠 面试官可能的“追问” (防御准备)

**追问 1：既然 useRef 不触发渲染，那用户怎么看到打字机效果？**
> **回答：** useRef 只是用来“暂存”极高频的数据。我依然会使用 State，只是我会控制 State 的更新频率。比如借助 `lodash/throttle` 或者 `setTimeout`，每 50 毫秒把当时 ref 里的最新字符串同步给 State。这样用户肉眼看依然是流畅的打字机，但 React 底层的压力小了非常多。

**追问 2：除了节流，React 18 有没有什么新特性可以处理这种高频更新？**
> **回答：** 有的。React 18 引入了 `useTransition` (并发渲染) 和自动批处理（Automatic Batching）。如果是更复杂的场景，我会考虑用 `startTransition` 把文本的更新标记为“低优先级”，保证用户在输入框打字（高优先级）不会被大模型的输出给卡住。不过在这个项目中，通过 Ref 缓冲已经达到了很好的性能指标。

**追问 3：为什么不用 Axios 处理流式请求？**
> **回答：** 虽然现在的 Axios 也支持了一些流式配置，但在浏览器环境下，处理 SSE 和 `ReadableStream` 时，原生的 `Fetch API` 提供的 `.body.getReader()` 心智模型更直接，控制粒度也更细。为了极致的性能和减少依赖包体积，我选择了原生 Fetch。
