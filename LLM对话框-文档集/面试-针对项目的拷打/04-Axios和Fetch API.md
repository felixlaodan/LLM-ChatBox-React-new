# 大白话讲解：为什么流式请求要用 Fetch 而不是 Axios？

要弄懂这个问题，我们先要明白一个核心比喻：**“提水桶”** 和 **“接水管”**。

## 1. 传统请求（提水桶） vs 流式请求（接水管）

- **传统请求（普通 API）**：就像是你要一桶水。你向服务器发请求，服务器在远端把一整桶水装满，然后一次性给你端过来。你必须等这桶水**完全装满**才能拿到手。
- **流式请求（打字机效果）**：就像是你接了一根水管。服务器只要产出一滴水，就立刻顺着水管流给你。你可以**边接边喝**，不用等全接满。

大模型生成文字非常慢（可能要几十秒），如果用“提水桶”的方式，用户就要盯着白屏看几十秒，体验极差。所以我们必须用“接水管”的方式，也就是**流式请求（Streaming）**。

---

## 2. 为什么 Axios 不适合做流式请求？

Axios 是我们平时开发最爱用的网络请求库，它有很多优点（自动转 JSON、拦截器等），但它在处理流式数据时，表现得像个**“强迫症的快递员”**。

### Axios 的底层原理与局限：

1. **底层机制**：在浏览器环境下，早期和主流版本的 Axios 底层封装的是老式的 `XMLHttpRequest` (XHR) 对象。XHR 天生就不是为了流式传输设计的。
2. **“强迫症”行为**：Axios 的设计初衷是为了让开发者更舒服。当你发起请求时，它会在后台**默默等服务器把所有数据都发完（水桶装满）**，然后帮你把数据解析成一个漂亮的 JSON 对象，最后才触发 `Promise.resolve(res)` 交给你。
3. **后果**：如果在大模型对话里用传统的 Axios 调 API，它会把流式返回的数据全部**攒在肚子里**。直到模型几十秒后说完最后一句话，Axios 才会把完整的一大段话吐给你。**打字机效果直接变成了“憋大招”效果。**

_(注：虽然 Axios 最新的版本在 Node.js 端支持流，在浏览器端也增加了一些对 Fetch 适配器的实验性支持，但在浏览器里直接用原生 Fetch 处理流，依然是业界最标准、最没有坑的做法。)_

---

## 3. 为什么 Fetch API 是流式请求的神器？

Fetch 是浏览器原生自带的现代网络请求 API。它之所以能完美胜任流式请求，是因为它的设计理念非常底层且先进：**它支持 `ReadableStream`（可读流）**。

### Fetch 的工作机制：

1. **响应极快**：当你用 `fetch()` 发起请求时，**只要服务器一返回响应头（Headers）**，Fetch 的 Promise 就会立刻变为成功状态（Resolved）。它**根本不等等身体（Body）数据传完**。
2. **暴露水管（ReadableStream）**：Fetch 会把尚未接收完的数据身体包装成一个叫 `response.body` 的对象。这就是那根“水管”，在代码里它叫 `ReadableStream`。
3. **提供水龙头（Reader）**：你可以从这根水管上接一个水龙头，叫做 `reader = response.body.getReader()`。有了水龙头，你就可以写一个循环，不断地拧开水龙头（`reader.read()`），只要网线里传来哪怕一个字节的数据（Chunk），你都能立刻接住并显示到屏幕上。

---

## 4. Fetch 实现流式请求的核心步骤（代码逻辑拆解）

为了让你更有底气，下面是 Fetch 接水的标准四步走（你能看懂这个逻辑，面试就稳了）：

```javascript
// 1. 发起 Fetch 请求
const response = await fetch('https://api.xxx.com/chat', { ... });

// 2. 接上水龙头（获取可读流的读取器）
const reader = response.body.getReader();

// 工具：因为网络传过来的是二进制（Uint8Array），我们需要一个翻译官把它变成文字
const decoder = new TextDecoder('utf-8');

// 3. 开始死循环，不断从水龙头接水
while (true) {
    // done 代表水端完了没，value 是这次接到的那一点点二进制水滴 (Chunk)
    const { done, value } = await reader.read();

    // 如果服务器说发完了，就打破循环退出
    if (done) {
        break;
    }

    // 4. 把二进制水滴翻译成文字，并更新到屏幕上
    const chunkText = decoder.decode(value, { stream: true });
    console.log("接到一滴水：", chunkText);

    // 这里你就会调用 Zustand 的 updateMessage 把文字追加到气泡里，形成打字效果
}
```

### 总结

简单来说：**Axios 默认喜欢“攒满一波带走”，而原生 Fetch API 暴露了底层的流式接口（ReadableStream），允许我们“来一点拿一点”**。这就是为什么实现大模型打字机效果时，必须抛弃 Axios（或者把 Axios 的底层完全魔改），直接拥抱原生 Fetch 的原因。

# 面试应对

**面试官：** “你是怎么实现流式请求的？既然平时项目里都用 Axios，为什么这个功能不用 Axios 而是直接用了 Fetch API？”

**你的回答逻辑（三步走，直切要害）：**

### 第一步：指出核心矛盾（直接指出 Axios 的短板）
> “在这个项目里，因为大模型生成文字非常慢，为了实现‘打字机’的体验，我们必须用流式（Streaming）的方式来接收数据。
>
> 平时我确实常用 Axios，它的拦截器和数据转换很好用。但是在浏览器环境下，早中期的 Axios 底层是基于 `XMLHttpRequest` (XHR) 实现的。XHR 的设计机制决定了，它倾向于**‘把服务器的数据全部接收完毕后’**，再一次性触发回调。这就导致如果用 Axios 请求大模型，不管你设置什么，它也会一直憋着，直到几十秒后模型把整段话说完才返回，完全破坏了打字机效果。”

### 第二步：解释为什么选择 Fetch（抛出 ReadableStream）
> “相对而言，浏览器原生的 **Fetch API** 更加现代。当我们发出 Fetch 请求时，只要服务器一返回响应头，Fetch 就能立刻把 `response` 对象交给我们，而不需要等待 Body 下载完。
> 
> 最关键的是，Fetch 暴露了底层的 **`ReadableStream`（可读流）** 接口。也就是 `response.body`。只要拿到了这个流，我就可以创建一个 Reader，通过 `reader.read()` 真正做到‘服务器发一点，我前端就读一点’。”

### 第三步：简述代码实现（展现手写代码能力）
> “所以我的具体实现是：
> 1. 用 `fetch` 发起 POST 请求。
> 2. 获取 `const reader = response.body.getReader()`。
> 3. 写一个 `while` 死循环，不断地 `await reader.read()` 获取返回的 Chunk（也就是二进制的 Uint8Array）。
> 4. 用 `TextDecoder` 把二进制流解码成文本，再处理成 JSON 数据。
> 5. 拿到解析出的单个 Token 后，再触发 Zustand 的状态更新，最终让页面上一个字一个字地蹦出来。”

**面试官可能追问（防守点）：**
1. **“那如果现在非让你用 Axios 怎么办？”**
   * *回答思路*：“其实 Axios 在 `v1.5.0` 之后增加了一个叫 `fetch` adapter 的实验性适配器，如果我们强行配置开启这个适配器并且指定 `responseType: 'stream'`，也许可以勉强做到。但是为了代码的稳定性和减少依赖黑盒，业界目前针对 AI 大模型的 SSE 流式请求，公认的最佳实践依然是直接使用原生 Fetch。”
2. **“那平时业务请求你也用 Fetch 吗？”**
   * *回答思路*：“不会。除了打字机流式接口之外，我的登录、获取历史记录等普通 API 依然使用的是 Axios，因为 Axios 在统一注入 Token 拦截、错误统一抛出方面，依然有着非常好的工程化体验。所以我是**按场景选技术**。”
