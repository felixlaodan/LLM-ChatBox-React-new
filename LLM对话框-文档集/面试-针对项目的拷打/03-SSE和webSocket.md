# 大白话讲解：SSE 与 WebSocket

在做这个大模型对话项目时，我们用到了“流式返回”（打字机效果）。要实现这个效果，网络协议层面有几种选择，最常被对比的就是 **SSE (Server-Sent Events)** 和 **WebSocket**。

为了让你彻底弄懂，我们从最基础的讲起。

## 1. 为什么需要这俩兄弟？（传统 HTTP 的痛点）

传统的 HTTP 请求就像是**写信**：

1. 客户端（你）写一封信发给服务器（后端）。
2. 服务器收到信，处理半天，然后写一封回信寄给你。
3. 信件一来一回，**必须是你先问，他才答**。而且每次问答，都要重新建一个连接（邮差跑一趟）。

**在大模型场景下的问题：**
大模型生成文字是很慢的。如果你问它“写一篇5000字的文章”，传统 HTTP 会让你干等1分钟，然后突然把5000字全塞给你。在这一分钟里，页面是死机的，用户体验极差。

我们需要一种技术，能让服务器“**挤牙膏**”一样，生成一点，就马上推给前端一点。

---

## 2. 什么是 WebSocket？（全双工、打电话）

WebSocket 就像是**打电话**：

1. 你拨通电话（建立连接）。
2. 只要电话不挂，**你可以随时说话，对方也可以随时说话**。
3. 双方是完全平等的，这叫**全双工 (Full-duplex)**。

**特点：**

- **双向通信**：客户端和服务器可以互相主动发消息。
- **协议不同**：它是一个独立的协议，虽然借用了 HTTP 来建立连接，但一旦连上，就变成 `ws://` 或 `wss://` 协议了。
- **数据格式**：既能传文本，也能传二进制文件。

**适用场景：**
多人在线游戏、实时聊天室（微信、QQ）、协同文档编辑（腾讯文档）。这些场景下，双方都需要频繁地主动发消息。

---

## 3. 什么是 SSE (Server-Sent Events)？（单工、听收音机）

SSE 就像是**听收音机**（或者关注公众号）：

1. 你打开收音机调到特定频道（发起普通的 HTTP 请求）。
2. 然后你就只能**单向接收**电台发来的广播，你不能对着收音机说话。
3. 这叫**单向通信 (Unidirectional)** 或 **半双工**。

**特点：**

- **单向通信**：**只能是服务器向客户端发消息**，客户端不能通过这个连接发消息给服务器（客户端想发消息，得另外发一个普通的 HTTP 请求）。
- **基于 HTTP**：它完全基于传统的 HTTP 协议（HTTP/1.1 或 HTTP/2）。只是在响应头里加了一句 `Content-Type: text/event-stream`，告诉浏览器：“别急着关连接，后面还有数据源源不断地来”。
- **只传文本**：它天生只支持纯文本数据（通常是 UTF-8 编码）。
- **内置断线重连**：浏览器原生支持（原生 `EventSource` API 支持），断网了会自动尝试重连。不过在咱们项目中，用 `fetch` 手写解析流，主要看重的是它底层的 HTTP 传输机制。

---

## 4. 终极拷问：为什么大模型对话普遍选 SSE 而不用 WebSocket？

这就是面试官最爱问的点。听起来 WebSocket 更牛逼（双向通信），为什么不用呢？

原因有以下几点（核心是**杀鸡焉用牛刀** + **场景极度契合**）：

### 理由一：场景需求契合（只需要单向数据流）

在 LLM 对话中，交互模式是：
前端发送一个 Prompt（普通 HTTP POST 请求） ---> 后端大模型开始推理 ---> 后端将生成的 Token **源源不断地单向推给前端**。
在这个推流的过程中，前端是不需要向后端发消息的（除非是发一个中止请求，那也是另起一个普通的请求）。**单向的 SSE 完美契合这种“请求 -> 流式响应”的模式。** WebSocket 的双向特性在这里显得多余。

### 理由二：实现成本低、轻量级

- **SSE** 走的是标准的 HTTP 请求。对于后端的网关、Nginx 代理、防火墙来说，它就是一个普通的 HTTP 请求，非常容易配置和穿透，不需要像 WebSocket 那样去处理复杂的协议升级（Upgrade）和特殊的代理配置。
- 在前端，发 SSE 就像发一个普通的 `fetch` 请求一样，只要拿到 `response.body.getReader()` 接管流的读取就能自己处理，非常轻量。

### 理由三：数据格式正好是文本

大模型吐出来的都是 Token（文本字符），SSE 天生就是为文本流设计的（`text/event-stream`），完全对口。而 WebSocket 为了支持二进制等各种格式，协议帧（Frame）的设计更复杂。

### 理由四：大厂 API 的事实标准

OpenAI、Anthropic 以及国内的硅基流动、通义千问等，它们提供的官方 API 接口，统统都是基于 HTTP 的 SSE 流式响应。既然上游就是 SSE，我们在前端直连时，继续保持 SSE 接收是最顺理成章、性能损耗最小的做法。

---

## 总结对比表格（面试脑图）

| 特性                | SSE (Server-Sent Events)    | WebSocket                    |
| :------------------ | :-------------------------- | :--------------------------- |
| **通信方向**        | **单向** (Server -> Client) | **双向** (Server <-> Client) |
| **底层协议**        | 纯 HTTP                     | 独立的 WebSocket 协议        |
| **数据格式**        | 仅支持纯文本 (UTF-8)        | 支持文本和二进制             |
| **代理/网关友好度** | **极高**（就是普通 HTTP）   | 较低（需要专门配置协议升级） |
| **实现复杂度**      | 极简                        | 较复杂                       |
| **适用大模型场景**  | **⭐⭐⭐⭐⭐ 完美契合**     | ⭐⭐ 可用但没必要（过重）    |

# 面试应对

**面试官：** “你这个项目中流式打字机效果是怎么做的？为什么用 SSE 而不用 WebSocket？”

**你的回答逻辑（三步走，层层递进）：**

### 第一步：一句话定性，讲清楚业务场景
> “在这个项目中，流式返回的本质是一个**单向的数据下发**场景。用户发送 Prompt 后，模型需要将生成的 Token 持续不断地推送给前端。因为这属于典型的 `Server -> Client` 单向推流，所以我选用了 SSE（Server-Sent Events）而不是 WebSocket。”

### 第二步：对比两种协议（踩一踩 WebSocket，捧一捧 SSE）
> “其实在做技术选型时我对比过这两种方案。
> 
> 虽然 WebSocket 是全双工的，支持双向通信，听起来更强大，但在 LLM 对话这种不需要前端频繁主动发消息的场景下，显得有些**‘杀鸡用牛刀’**。而且 WebSocket 协议相对较重，在网关代理（比如 Nginx）配置上也更复杂，需要专门处理协议升级（Upgrade）。
> 
> 相比之下，**SSE 完全基于标准的 HTTP 协议**，轻量且天生对防火墙和网关友好。它只需要在响应头加上 `text/event-stream`，而且正好原生只支持文本传输，完美契合大模型返回 Token 的需求。”

### 第三步：落地到项目代码（展示你的实操能力）
> “在具体实现上，我没有使用原生的 `EventSource` 对象，因为原生 API 在发送 POST 请求和携带鉴权 Token 时不太方便。
> 
> 结合业界标准的 OpenAI 接口规范，我直接使用了原生 **Fetch API 的可读流（ReadableStream）**。通过 `response.body.getReader()`，再配合 `TextDecoder` 和一个 `while` 循环，自己手动解析 Chunk 并处理可能出现的**‘粘包拆包’**（截断）问题，最后对接 Zustand 更新状态，实现了高性能的流式渲染。”

**面试官大概率会追问（防守点）：**
1. **“你刚刚提到的‘粘包拆包/截断问题’，能具体讲讲吗？”**
   * *回答思路*：TCP传输不可控，一个 Chunk 可能并不是一个完整的 JSON。我通过维护一个 Buffer（缓存字符串），按换行符 `\n\n` 分割，遇到解析失败的不完整包就留在 Buffer 里，等下一个 Chunk 来了拼上再解析。
2. **“怎么主动中断这个流？”**
   * *回答思路*：使用 `AbortController`。在发出 Fetch 请求时传入 signal，点击“停止生成”或组件卸载时调用 `controller.abort()` 关闭连接。
