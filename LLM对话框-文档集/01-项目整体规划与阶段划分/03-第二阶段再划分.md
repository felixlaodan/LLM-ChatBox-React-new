### 🧠 第二阶段：状态管理与逻辑核心 (详细拆解)

#### 2.1 小阶段：类型定义与设置仓库 (Types & Settings Store)
**目标**：建立全局类型系统，并实现用户设置（API Key、模型参数）的持久化存储。

*   **📝 要做的事情**：
    1.  **定义核心类型 (`src/types/index.ts`)**：
        *   定义 `Message` (角色, 内容, 时间戳)。
        *   定义 `Conversation` (ID, 标题, 消息列表)。
        *   定义 `Settings` (API Key, Model, Temperature 等)。
    2.  **创建设置仓库 (`src/store/useSettingsStore.ts`)**：
        *   使用 `zustand` 创建 Store。
        *   使用 `persist` 中间件（实现自动保存到 localStorage）。
        *   实现 `updateSettings` 和 `resetSettings` 方法。

*   **✅ 验收标准**：
    *   在浏览器控制台 (Console) 中调用 `useSettingsStore.getState().updateSettings({ apiKey: '123' })`。
    *   **刷新页面**后，再次打印 Store，`apiKey` 依然是 `'123'`（证明持久化成功）。

---

#### 2.2 小阶段：聊天数据仓库 (Chat Store - CRUD)
**目标**：实现会话管理的增删改查逻辑。这是最复杂的数据逻辑部分。

*   **📝 要做的事情**：
    1.  **创建聊天仓库 (`src/store/useChatStore.ts`)**：
        *   定义 `conversations` 数组和 `currentConversationId`。
    2.  **实现核心 Action**：
        *   `createConversation()`: 新建一个空会话并自动切换过去。
        *   `deleteConversation(id)`: 删除会话，并智能切换到下一个会话。
        *   `addMessage(message)`: 向当前会话追加一条消息。
        *   `updateLastMessage(content)`: **关键**，用于流式更新最后一条消息的内容（打字机效果）。
    3.  **集成 Persist**：确保聊天记录刷新不丢失。

*   **✅ 验收标准**：
    *   可以新建多个会话，Store 中数组长度增加。
    *   可以切换当前会话 ID。
    *   向会话 A 发送消息，切换到会话 B 再切回来，会话 A 的消息还在。

---

#### 2.3 小阶段：API 服务层与流式处理 (API Service & Streaming)
**目标**：封装网络请求，攻克“流式响应”这一技术难点。

*   **📝 要做的事情**：
    1.  **封装 API 客户端 (`src/lib/api.ts`)**：
        *   虽然我们用 Axios，但对于 **SSE (Server-Sent Events)** 流式响应，原生 `fetch` 配合 `ReadableStream` 往往更可控。我们需要编写一个 `streamChatCompletion` 函数。
        *   实现 API Key 的自动注入（从 Settings Store 获取）。
    2.  **解析流数据**：
        *   编写解析器，处理 `data: { "content": "..." }` 格式的数据包。
        *   处理异常情况（如 API Key 错误、网络断开）。

*   **✅ 验收标准**：
    *   编写一个简单的测试脚本或临时按钮。
    *   点击后，能在控制台看到 API 返回的文本是**一段一段**打印出来的（而不是一次性出来）。
    *   能正确捕获 401 (未授权) 等错误。

---

#### 2.4 小阶段：逻辑联调 (Integration "Dry Run")
**目标**：将 Store 和 API 串联起来，验证完整的数据闭环。

*   **📝 要做的事情**：
    1.  **编写 `useSubmitMessage` Hook**：
        *   这是一个自定义 Hook，负责协调：用户点击发送 -> 添加 User 消息 -> 调用 API -> 接收流 -> 实时更新 Assistant 消息 -> 保存。
    2.  **创建临时测试组件 (`src/components/DebugPanel.tsx`)**：
        *   放一个简单的 Input 和 Button。
        *   绑定上面的 Hook。
        *   用 `<pre>` 标签把 Store 里的当前对话 `JSON.stringify` 出来显示在页面上。

*   **✅ 验收标准 (最终呈现结果)**：
    *   在丑陋的测试页面上，输入 "Hello"。
    *   点击发送。
    *   页面上的 JSON 数据实时跳动，看到 "Hello" 被加入。
    *   紧接着看到 "Assistant" 的消息从空字符串 `""` 逐渐变成 `"Hi"` -> `"Hi there"` -> `"Hi there!"`。
    *   整个过程无报错，数据结构完整。
